# День 1 — Rules: разница двух генераций

Боевая задача (один промпт, без уточнений и переписок, для обеих генераций):
> Напиши новый модуль `notify.py` для task33: по `ticket_id` собрать сводку тикета
> (id, статус, тариф/роль юзера, код ошибки, последнее сообщение), POST на webhook из
> `NOTIFY_WEBHOOK`, при пустом URL — печать в stdout. CLI `python notify.py T-1042`.

| | **gen1 — правила v1 (тонкие)** | **gen2 — правила v2 (полные)** |
|---|---|---|
| Файл | `gen1_notify.py`, 127 строк | `gen2_notify.py`, 116 строк |
| `py_compile` | ✅ OK | ✅ OK |
| HTTP | stdlib `urllib.request` | stdlib `urllib.request` |
| Docstring | русский, с пайплайном | русский, с пайплайном + ссылка на §5.1 |
| Type hints | есть везде | есть везде |
| Константы | `WEBHOOK`, `TIMEOUT` | `WEBHOOK`, `TIMEOUT`, `_DASH` |
| Порядок файла §8 | соблюдён | соблюдён + явные ссылки на § в комментах |
| Сводка | плоский dict → рендер | сразу текст + `subject` тикета в заголовке |
| Возврат `notify()` | сама сводка (dict) | канал доставки `'webhook'/'stdout'` |
| Обработка отсутствующего юзера | нет (пробьётся в except CLI) | `_safe_user()` ловит `NotFound` → `None`, прочерк |
| **Битый ticket_id (`T-9999`)** | ✅ **аккуратно**: `except CRMError` в `main()` → сообщение + exit 1 | ❌ **traceback**: `get_ticket` не обёрнут, ошибка вылетает наверх |

## Главный вывод (что важнее файла правил)

**Оба варианта скомпилировались и оба в едином стиле проекта.** Дельта по стилю
оказалась тоньше ожидаемой: базовая модель сильная и, прочитав один существующий
файл проекта (`crm_tools.py`), сама скопировала домашние конвенции — stdlib, русский
docstring, разделители, `from __future__`. Правила v1 «повезло», правила v2 —
**гарантировали** тот же результат без везения и добавили осмысленные детали
(канал доставки, `subject`, безопасный доступ к юзеру, ссылки на §).

НО: **правила управляют стилем и консистентностью, а не корректностью.** Именно
«правильный по правилам» gen2 упал стектрейсом на плохом вводе, а «тонкий» gen1 —
нет. Это ключевой урок дня: rules-файл ≠ замена тестам.

## Что сильнее всего повлияло на качество (по убыванию)

1. **Наличие в репозитории эталонного файла для чтения** (`crm_tools.py`).
   Один живой пример дал модели ~80% стиля ещё до всяких правил. Правила без
   примеров работают слабее, чем правила + образец кода рядом.
2. **Явные примеры «как ХОТИМ» (§6) и антипаттерны (§7)** — убрали разброс:
   гарантированный `from __future__`, порядок файла, схема метаданных.
3. **Требование каскада с деградацией (§5.1)** — обе генерации сделали fallback
   на stdout при пустом/мёртвом webhook.
4. **Шаблон файла (§8)** — задал предсказуемую раскладку (docstring→импорты→
   константы→API→хелперы→main).

## Итерация правил v2 → v2.1 (доработка по результатам)

Провал gen2 на битом id обнажил дыру: **ни v1, ни v2 не требовали ловить доменные
ошибки на границе CLI**. Добавлено в `GIT/CLAUDE.md`:
- §7.11 — антипаттерн «непойманная доменная ошибка на границе CLI»;
- §10 — пункт Definition of Done: прогнать на плохом вводе, ожидать понятную
  ошибку + exit 1, а не traceback.

С правилом v2.1 корректная генерация обязана обернуть весь доменный вызов в
`try/except crm_tools.CRMError` в `main()` — как это уже (случайно) сделал gen1.
