# Видео-сценарий — День 3. Testing (5–7 мин)

Цель ролика: показать два уровня тестов, как агент сам прогоняет UI и как
код-тест ловит реальный баг проекта.

---

## 0. Вступление (0:00–0:40)
Голос: «День 3 курса — тестирование. Делаю два уровня. Первый — код-тесты
на бизнес-логику проекта ТН ВЭД: роутер, чанкинг, метрики, реестр инструментов.
Второй — UI-smoke: агент сам открывает страницы, кликает, заполняет формы и
проверяет результат через Playwright. В конце — единый прогон для PR.»

Показать дерево: `tree day03-testing -L 2` (или `ls -R`).

## 1. Level 1 — код-тесты (0:40–2:30)
Команда:
```bash
.venv/bin/python -m pytest -v
```
Голос: «38 тестов на четыре реальных модуля проекта, не на копии — `conftest.py`
подключает `task35/` и `task33/` напрямую. Всё зелёное с первого запуска.»

Открыть `level1_code/test_router.py` — показать 2–3 кейса (код с пробелами →
CODE_LOOKUP, обычный текст → FULL_PIPELINE, callback-приоритет).

## 2. Тест поймал баг (2:30–3:40)
Открыть `level1_code/test_router_bug.py`.
Голос: «Пока писал тесты — нашёл дефект. Голый десятизначный код `8525893000`
без пробелов роутер НЕ распознаёт: в regex `{9,12}` минимум одиннадцать символов,
а код — десять. Из-за этого запрос уходит в дорогой полный пайплайн вместо
быстрого. Фикс — одна цифра, `{9,12}` на `{8,12}`. Тест помечен xfail: сюит
зелёный, но баг зафиксирован с точкой в коде и готовым фиксом.»

Показать в pytest-выводе строку `XFAIL ... router.py:19 ... фикс -> {8,12}`.

## 3. Level 2 — UI smoke (3:40–5:40)
Команда:
```bash
.venv/bin/python level2_smoke/smoke_runner.py
```
Голос: «Мишень — маленькое CRUD-приложение "Заявки ТН ВЭД": логин, создать,
проверить, удалить. Раннер сам поднимает сервер, гоняет пять сценариев в
Playwright и на каждый шаг делает скриншот.»

Показать вывод `✅ S1…S5`, затем открыть `level2_smoke/SMOKE_REPORT.md` и папку
`screenshots/` — пролистать 2–3 png (список из двух заявок, состояние после
удаления, ошибка неверного пароля).

Открыть `scenarios.md` — показать, что сценарии описаны и текстом (S1–S5).

## 4. Единый flow для PR (5:40–6:40)
Команда:
```bash
bash flow/run_all.sh "$(date '+%F %T')"
```
Показать `REPORT.md`: оба уровня, вердикт «можно мержить», exit-код 0.
Открыть `flow/POST_PR_FLOW.md` — YAML для GitHub Actions + сценарий
«задеплоил фичу → обнови smoke → прогони всё».

## 5. Вывод (6:40–7:00)
Голос: «Итог: код-тесты на четыре модуля, пять smoke-сценариев, которые агент
протыкивает сам, отчёты со скриншотами и один прогон на PR-гейте. Плюс тесты
сразу окупились — поймали реальный баг роутера.»
