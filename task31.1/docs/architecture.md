# Архитектура проекта deepseek-chat

## Слои

1. **CLI-слой** — `cli.py`, точка входа для диалога с DeepSeek. Держит
   историю сообщений в памяти и режимы ответа (free/strict/nano).
2. **LLM-слой** — два провайдера:
   - DeepSeek API (облако) через HTTP, ключ в env `DEEPSEEK_API_KEY`.
   - Ollama + Qwen3 14B (локально, порт :11434) — для офлайн-инференса.
3. **RAG-слой** — пайплайн ретривала:
   - чанкинг документов,
   - эмбеддинги (GigaEmbeddings, 2048 dim),
   - хранение в Qdrant (коллекция `youtube_rag`, ~1792 вектора),
   - реранкинг (threshold + MMR), цитаты.
4. **MCP-слой** — серверы Model Context Protocol дают модели инструменты:
   доступ к CouchDB (task17), к git-контексту (task32) и т.д. Транспорт —
   stdio или HTTP через nginx.
5. **Агентный слой** — YAML-профили агентов под разные задачи (task31).

## Поток данных RAG

```
документ → chunking.py → чанки+метаданные → indexer.py → эмбеддинги → индекс
запрос → эмбеддинг запроса → cosine similarity → топ-K чанков → LLM → ответ
```

## Метаданные чанка

Каждый чанк несёт: `source` (файл), `title` (заголовок документа),
`section` (заголовок секции), `chunk_id` (порядковый номер).

## Инфраструктура

- Домашняя машина — индексация и torch (`~/torch-env/`).
- NL №1 (VPS) — MCP-сервисы, общий venv `/opt/mcp-venv/`.
- Один shared venv на все MCP-сервисы: новые зависимости ставятся туда.

## Ассистент разработчика (task32)

Отдельный слой поверх репозитория: RAG над README+docs проекта плюс живой
git-контекст через MCP. Команда `/help` отвечает на вопросы о структуре
проекта, объединяя найденные фрагменты документации и текущее состояние git
(ветка, изменённые файлы, diff).
