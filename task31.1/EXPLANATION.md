# Объяснение task32 — как для ребёнка

## Что мы сделали (одним абзацем)

Мы сделали «умного помощника программиста», который знает наш проект. Представь
нового сотрудника, которому дали две вещи: **папку с инструкциями** (README и
docs) и **пропуск в мастерскую**, чтобы он мог зайти и посмотреть, что там
происходит прямо сейчас. Инструкции он выучил наизусть (это RAG), а через пропуск
видит живую обстановку — на какой «полке» (ветке) сейчас работают, что поменяли
(git). Когда ты спрашиваешь «а как тут всё устроено?» — он смотрит и в
инструкции, и в мастерскую, и отвечает. Команда для вопроса называется `/help`.

---

## Каждый термин — простыми словами

| Термин | Что это | Пример из жизни |
|---|---|---|
| **Ассистент разработчика** | Программа, отвечающая на вопросы о проекте | Экскурсовод в музее: спросил — рассказал |
| **Документация (README + docs)** | Текст, объясняющий, как устроен проект | Инструкция к конструктору LEGO |
| **RAG** | «Найди нужный кусок текста и ответь по нему» | Ты не помнишь всю энциклопедию — открываешь нужную страницу и читаешь |
| **Чанкинг** | Разрезать документ на кусочки | Разрезать пиццу на ломтики, чтобы удобно брать |
| **Эмбеддинг** | Превратить текст в числа, чтобы сравнивать по смыслу | Каждой песне дать «отпечаток», чтобы находить похожие |
| **TF-IDF** | Простой способ измерить, о чём кусок текста | Подсчёт, какие слова в тексте самые «говорящие» |
| **Косинусная близость** | Насколько два текста похожи | Два человека идут в одну сторону = похожи; в разные = нет |
| **Индекс (index.json)** | Файл со всеми кусочками и их метками | Оглавление книги: где что искать |
| **MCP** | Стандарт, как дать программе «инструменты» | Розетка единого стандарта: любой прибор воткнёшь |
| **git-ветка** | На какой версии проекта сейчас работают | Развилка дороги: пошли по этой тропинке |
| **git diff** | Что изменилось с прошлого раза | Игра «найди 10 отличий» между двумя картинками |
| **/help** | Команда «ответь на мой вопрос» | Нажал кнопку «справка» и получил ответ |
| **Fallback (запасной путь)** | Если главное не работает — есть замена | Нет лифта — идёшь по лестнице, всё равно поднимешься |

## Как это работает по шагам

1. **Готовим знания.** `chunking.py` режет README и docs на кусочки по
   заголовкам. `indexer.py` складывает их в `index.json` с метками (какой файл,
   какая секция).
2. **Учимся искать.** `search.py` превращает кусочки в числа (TF-IDF) и по
   вопросу находит самые похожие — топ-4.
3. **Смотрим в мастерскую.** `mcp_server.py` — это «пропуск»: по команде отдаёт
   ветку, файлы, diff. `git_tools.py` — руки, которые реально дёргают git.
4. **Отвечаем.** `assistant.py` берёт найденные кусочки + git-контекст и делает
   ответ. Если есть ключ DeepSeek — красиво формулирует нейросеть; нет — отвечает
   Ollama; нет и его — просто показывает найденные кусочки. Ответ будет всегда.

## Почему именно так (обоснование решений)

- **RAG без векторной БД (JSON-индекс).** Документов мало (4 файла). Поднимать
  Qdrant ради 24 кусочков — как арендовать грузовик, чтобы отвезти один пакет.
  JSON-файл проще и работает офлайн.
- **TF-IDF по умолчанию.** Он на чистом Python, без установки библиотек. Значит,
  задание запускается на любой машине сразу. Умные эмбеддинги (sentence-
  transformers) оставили как опцию — включаются одной переменной.
- **MCP своими руками на stdlib.** Официальный SDK требует установки. Мы написали
  маленький сервер, который говорит на том же «языке» (JSON-RPC), — и он
  запускается на голом python3. Честный протокол, ноль зависимостей.
- **Три пути генерации ответа.** Инфраструктура (Ollama, ключи) может быть
  выключена. Чтобы ассистент не «умирал», сделали лесенку: DeepSeek → Ollama →
  просто фрагменты. Помощник отвечает при любой погоде.
- **Общий `git_tools.py`.** Одна логика git на всех — не дублируем код между
  MCP-сервером и ассистентом. Поправил в одном месте — работает везде.

## Что ещё могло быть (альтернативы)

| Решение | Плюсы | Минусы |
|---|---|---|
| **Qdrant вместо JSON** | Масштабируется на тысячи документов | Нужен запущенный сервер, тяжело для 4 файлов |
| **FAISS-индекс** | Быстрый поиск на больших объёмах | Лишняя зависимость, пусто для 24 чанков |
| **Только dense-эмбеддинги** | Понимают смысл, не только слова | Требуют скачать модель, не работает офлайн |
| **Официальный MCP SDK** | Полный протокол «из коробки» | Установка пакетов, тяжелее демонстрировать |
| **Ответ только через LLM** | Красивые формулировки | Падает, если LLM недоступна — а она была выключена |

Мы выбрали лёгкие варианты, потому что цель задания — показать связку
**документация → RAG → MCP → /help**, а не построить продакшн-систему.

---

## Итог одной фразой

Мы собрали помощника, который выучил инструкции проекта (RAG над README+docs),
подглядывает в живой git через стандартный «переходник» MCP и по команде `/help`
отвечает на вопросы о том, как проект устроен, — и делает это на голом python3,
без единого внешнего сервиса.
