# Объяснение — День 24: Цитаты, источники и анти-галлюцинации

---

## Что сделали (одним абзацем)

Научили RAG-агент всегда показывать откуда он берёт информацию. Теперь каждый ответ содержит: список источников с их идентификаторами, прямые цитаты из найденных документов, и механизм отказа — если база знаний не покрывает вопрос, агент честно говорит "не знаю" вместо того чтобы придумывать.

**Аналогия:** Как если бы консультант не просто давал совет, а обязательно показывал страницу в законе на которую ссылается — и дословно её зачитывал.

---

## Термины

### Цитата (citation)
Дословный фрагмент из найденного документа, взятый в кавычки.

**Пример из жизни:** Судья цитирует статью закона дословно — нельзя "пересказать своими словами", только точный текст.

**Зачем в RAG:** Позволяет проверить не выдумал ли ответ модель. Если цитата есть в базе — ответ подтверждён. Если нет — галлюцинация.

---

### Источник (source)
Ссылка на конкретный документ/видео + идентификатор чанка + score релевантности.

**Пример из жизни:** Сноска в научной статье — [1] Иванов, 2023, с. 47. Не просто "я где-то читал".

**Зачем:** Прослеживаемость. Можно открыть исходный документ и убедиться что информация там действительно есть.

---

### Порог релевантности / "не знаю"
Минимальный cosine similarity (0.50), ниже которого агент отказывается отвечать.

**Пример из жизни:** Врач говорит "я не специалист в этом, обратитесь к кардиологу" — вместо того чтобы угадывать диагноз.

**Зачем:** Галлюцинация опаснее пробела. Если база знаний не покрывает вопрос — лучше честный отказ чем уверенный неправильный ответ.

---

### Семантическое совпадение (semantic match)
Метрика: какой процент значимых слов из ответа встречается в цитатах.

**Пример из жизни:** Если ответ говорит "налог на добавленную стоимость" а в цитатах есть "НДС" — слова не совпадут дословно, но смысл тот же. Простая метрика это не поймёт, для этого нужны эмбеддинги.

**Зачем 30% это нормально:** Модель перефразирует цитаты в ответе — это хорошо. Дословное совпадение было бы признаком что модель просто копирует текст без понимания.

---

## Почему именно так (технические решения)

### Структура промпта вместо парсинга JSON
Можно было попросить модель вернуть JSON `{"answer":..., "sources":..., "citations":...}`.

**Почему секции лучше:**
| JSON | Секции (ОТВЕТ:/ИСТОЧНИКИ:/ЦИТАТЫ:) |
|------|------------------------------------|
| Модель часто ломает JSON-синтаксис | Текстовые маркеры надёжнее |
| Требует JSON-парсера с обработкой ошибок | Regex из 3 строк |
| Модель путается с экранированием кавычек внутри JSON | Нет конфликта |

### think: False для Qwen3
Qwen3 14B по умолчанию "думает" через скрытые `<think>...</think>` блоки. Без `think: False` эти токены съедают весь лимит `num_predict` и модель не успевает ответить — возвращает пустую строку.

### Двойной детектор "не знаю"
1. Числовой порог (score < 0.5) — работает до LLM-вызова, экономит время
2. Текстовый детектор ("не знаю" в ответе) — ловит случаи когда score прошёл порог, но модель сама поняла что контекст нерелевантен

## Что ещё могло быть

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **Структурный промпт (наш выбор)** | Надёжно, просто, без лишних зависимостей | Парсинг может сломаться если модель меняет формат |
| **JSON output** | Строгая структура | Модели часто ломают JSON, нужен retry-логика |
| **Retrieval с reranking (task23)** | Точнее находит источники | Усложняет пайплайн, медленнее |
| **Confidence score от LLM** | Модель сама оценивает уверенность | Ненадёжно — модели переоценивают уверенность |
| **Hallucination detector (отдельная модель)** | Специализированная проверка | Нужна дополнительная модель, медленно |

---

## Итог в одной фразе

Принудили модель всегда показывать источники и цитаты через структуру промпта, и добавили честный отказ когда база знаний не покрывает вопрос — теперь каждый ответ проверяем, не доверяем на слово.
