# Прогон профиля Bug Fix — реальный баг

**Задача (как дал пользователь):**
> `[BUGFIX]` В task33 ответы Ollama обрываются / теряют часть RAG-контекста,
> будто модель не видит весь промпт. В task34 такого нет.

Ниже — ответ ассистента строго в формате профиля Bug Fix.

---

🔴 **БАГ:** Ollama-ветка в task33 молча теряет длинный промпт — qwen3 отвечает так,
будто не видел RAG/CRM-контекст.

🔬 **ВОСПРОИЗВЕДЕНИЕ:** промпт собирается в `_prompt()` — это ВОПРОС + CRM-контекст +
до 3000 символов RAG (`assistant.py:108–123`). Итого легко >4000 токенов. Запрос в
Ollama (`gen_ollama`, было `:148–152`) уходил телом:
```json
{"model": "qwen3:14b", "prompt": "...", "stream": false}
```
Без `options.num_ctx`. Дефолтное окно qwen3 в Ollama ~4k токенов → всё сверх окна
тихо отбрасывается на стороне сервера, без ошибки. Симптом = «модель игнорит часть
контекста», а не падение — потому и незаметно.

🎯 **ПРИЧИНА:** отсутствует `options.num_ctx` в теле запроса `/api/generate`.
Корень — `task33/assistant.py`, функция `gen_ollama`. Эталон рядом: `task34/llm.py:87`
уже шлёт `"options": {"temperature": ..., "num_ctx": OLLAMA_NUM_CTX}` (там `OLLAMA_NUM_CTX = 32768`).
Значит это не новый баг, а **непортированный фикс** — task33 старше.

🔧 **ФИКС:** порт эталона из task34, минимально.
```diff
  RAG_MIN_SCORE = 0.02  # ниже этого чанк считаем шумом
  RAG_TOP_K = 4
+ OLLAMA_NUM_CTX = 32768  # дефолт qwen3 ~4k → длинный prompt (RAG+CRM) молча режется

  def gen_ollama(prompt: str) -> str | None:
      body = json.dumps({"model": os.getenv("OLLAMA_MODEL", "qwen3:14b"),
-                        "prompt": prompt, "stream": False}).encode()
+                        "prompt": prompt, "stream": False,
+                        "options": {"num_ctx": OLLAMA_NUM_CTX}}).encode()
```
Константа — вверху файла с обоснованием (CLAUDE.md §4). Правка ровно одна, поведение
DeepSeek/offline-веток не тронуто.

✅ **ПРОВЕРКА:**
- `python -m py_compile task33/assistant.py` → чисто.
- Тело запроса теперь содержит `options.num_ctx` (проверено парсингом файла).
- Каскад не сломан: `gen_deepseek → gen_ollama → gen_offline` (`answer()`, `:351–362`)
  не менялся; при отсутствии Ollama по-прежнему `None` → офлайн-ветка (§5.1 соблюдён).

⚠️ **НЕ ПРОВЕРЕНО:** живой ответ qwen3 на длинном промпте — нужен поднятый
`ollama serve` с `qwen3:14b` (в этом окружении не поднят). Проверено на уровне
контракта запроса, не на уровне генерации. Это ожидаемо: цель фикса — чтобы Ollama
получала полное окно; сам факт передачи `num_ctx` подтверждён.

👯 **БЛИЗНЕЦЫ:** прогнал `grep -rn "api/generate\|api/chat\|num_ctx" task33 task34`.
В task34 уже правильно. Других мест с `/api/generate` без `num_ctx` в этих тасках нет.
Стоит проверить task31/32/35, если там тоже есть Ollama-ветка (вне скоупа этого бага).
