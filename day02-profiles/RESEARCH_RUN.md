# Прогон профиля Research — реальный вопрос

**Задача (как дал пользователь):**
> `[RESEARCH]` Как у нас в task33 устроен LLM-каскад — какие уровни, в каком
> порядке, что происходит, если ничего не доступно?

Ниже — ответ ассистента строго в формате профиля Research. Код не менялся (read-only).

---

❓ **ВОПРОС:** порядок и логика деградации LLM-каскада в ассистенте поддержки task33.

📂 **ФАЙЛЫ:**
- `task33/assistant.py:340–362` — `answer()`, оркестратор каскада (главная точка).
- `task33/assistant.py:126–143` — `gen_deepseek()`, уровень 1 (внешний API).
- `task33/assistant.py:146–157` — `gen_ollama()`, уровень 2 (локальная LLM).
- `task33/assistant.py:295–338` — `gen_offline()` + `offline_rules():192` — уровень 3 (правила+RAG, без сети).
- `task33/assistant.py:108–123` — `_prompt()`, сборка единого промпта (ВОПРОС+CRM+RAG) для уровней 1–2.

🔗 **СВЯЗИ:**
```
answer()                         # :340
  ├─ resolve_context()           # :346  CRM: тикет + клиент
  ├─ retrieve() → format_rag()   # :348  RAG-контекст (топ-K чанков)
  ├─ _prompt(question,ctx,rag)   # :353  один промпт на уровни 1–2
  ├─ gen_deepseek(prompt)  ──┐   # :354  уровень 1
  ├─ gen_ollama(prompt)    ──┤   # :358  уровень 2 (если 1 вернул None)
  └─ gen_offline(...)      ──┘   # :362  уровень 3 (если body всё ещё None)
```
Ключ каскада — переменная `body`: инициализируется `None` (`:351`), каждый уровень
пытается её заполнить; следующий вызывается только если предыдущий вернул `None`.

🧭 **РАЗБОР:**
1. **Уровень 1 — DeepSeek API** (`:354`). `gen_deepseek` проверяет `DEEPSEEK_API_KEY`
   (`:128–130`): нет ключа или плейсхолдер → сразу `None`, без сети. Сетевая/HTTP-ошибка
   ловится `except Exception` (`:141`) → печать диагностики + `None`. Успех → `body="...", gen="deepseek"`.
2. **Уровень 2 — Ollama** (`:358`) вызывается ТОЛЬКО если уровень 1 дал `None`.
   `gen_ollama` бьёт в `localhost:11434/api/generate` (`:150`). Не поднята → `except` → `None` (`:156`).
   *(здесь и был баг num_ctx — см. BUGFIX_RUN.md; на структуру каскада не влияет.)*
3. **Уровень 3 — офлайн** (`:361–362`). Если `body is None` после 1 и 2 — `gen_offline`
   собирает ответ из детерминированных правил (`offline_rules:192`, тематика auth/api/other
   по `_topic:169`) + RAG-хитов. Сеть и ключ не нужны. Этот уровень **не может вернуть None** —
   гарантирует непустой ответ.
4. Флаг `use_llm=False` (`:352`) полностью пропускает уровни 1–2 → сразу офлайн.

📌 **ВЫВОД:**
- Каскад: **DeepSeek → Ollama → офлайн**, строго по деградации, переход только на `None`.
- Соответствует правилу проекта §5.1 (`GIT/CLAUDE.md`): ответ содержателен всегда,
  каждый уровень ловит свою ошибку и падает на следующий, не роняя процесс.
- Офлайн-ветка — «дно» каскада: работает без сети и ключа, всегда даёт ответ.
- Единая точка сборки промпта (`_prompt`) → уровни 1–2 получают идентичный вход.

🎚️ **УВЕРЕННОСТЬ:**
- Порядок уровней и условия перехода — **подтверждено кодом** (`answer():351–362`).
- «Офлайн не возвращает None» — **подтверждено**: `gen_offline` всегда строит строку (`:295–338`).
- Поведение живого DeepSeek/Ollama при валидном ключе — **гипотеза** (в окружении не проверялось,
  разобрано по коду, не по запуску).
