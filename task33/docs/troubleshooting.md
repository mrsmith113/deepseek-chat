# Юко-Кабинет — разбор инцидентов

Справочник для поддержки: код ошибки → что это → что проверить → как починить.

## Таблица кодов

| Код | Что означает | Область |
|-----|--------------|---------|
| `AUTH-001` | Неверный логин или пароль | auth |
| `AUTH-002` | 2FA-код просрочен или неверен | auth |
| `AUTH-005` | Аккаунт временно заблокирован (5 неудачных попыток → 15 минут) | auth |
| `SSO-003` | SSO-домен не верифицирован (нет подтверждённой DNS TXT-записи) | auth |
| `API-401` | API-ключ отозван, неверен или из другой среды | api |
| `API-403` | Ключ валиден, но прав/тарифа недостаточно | api |
| `API-429` | Превышен rate limit | api |
| `WH-005` | Эндпоинт вебхука в статусе `failing` | webhooks |
| `BILL-002` | Платёж отклонён банком | billing |
| `LIM-001` | Исчерпан лимит заявок тарифа | billing |

## AUTH-001 — неверные креды

**Симптом:** «Неверный email или пароль» при каждой попытке.

**Проверить:**
1. Email тот же, что в приглашении (частая ошибка — личный вместо корпоративного).
2. Пользователь существует и активен: Команда → статус не `deactivated`.
3. Пользователь создан через SSO/SCIM — **у него нет пароля вообще**. Вход по паролю невозможен, пока не задан через «Забыли пароль?».
4. Пробел/перенос строки при вставке пароля из менеджера.

**Чинить:** «Забыли пароль?» → письмо (живёт 30 мин, проверить спам). Не приходит — проверить, не режет ли корпоративный почтовый фильтр домен `notifikat.ru`, попросить админа почты внести в allowlist.

**Эскалация:** если письмо не доставляется третий раз — тикет на инфраструктуру с message-id.

## AUTH-002 — 2FA-код просрочен

**Симптом:** пароль принят, код TOTP отбивается. Иногда «то работает, то нет».

**Проверить:**
1. Код вводится в текущем 30-секундном окне (не тот, что был на экране минуту назад).
2. **Расхождение времени на телефоне** — причина №1. Настройки телефона → дата и время → автоматически. Допуск сервера ±30 сек.
3. Приложение привязано к тому же аккаунту (у людей 3–4 записи «Юко» после перепривязок — удалить лишние).
4. Резервный код вводится без дефисов, каждый одноразовый.

**Чинить:** синхронизировать время → повторить. Не помогло — сброс 2FA админом (Команда → пользователь → «Сбросить 2FA») и повторная привязка.

## AUTH-005 — аккаунт временно заблокирован

**Симптом:** «Слишком много попыток входа. Попробуйте позже» — даже с верным паролем.

**Механика:** 5 неудачных попыток подряд → блокировка **на 15 минут**, снимается сама. Считаются попытки по email+IP. Успешный вход обнуляет счётчик.

**Проверить:**
1. Прошло ли 15 минут с последней попытки — каждая новая попытка **продлевает** окно, надо просто перестать долбить.
2. Не бьётся ли в аккаунт скрипт/интеграция со старым паролем (после смены пароля забыли обновить в 1С/скрипте) — тогда блокировка бесконечная. Смотреть audit log: источник попыток.
3. Не брутфорс ли извне — сравнить IP в audit log с офисным.

**Чинить:** подождать 15 минут без попыток. Если долбит интеграция — найти и остановить. Ускорить снятие: успешный сброс пароля через письмо снимает блокировку сразу.

## SSO-003 — домен не верифицирован

**Симптом:** Enterprise, `sso_enabled: true`, но кнопка «Войти через SSO» не работает / IdP редиректит с ошибкой. Пользователи говорят «не работает авторизация».

**Механика:** SAML не активируется, пока домен компании не подтверждён DNS TXT-записью. Пока `domain_verified: false` — SSO выключен принудительно, **работает только вход по логину и паролю**.

**Проверить:**
1. Тариф = Enterprise (на Pro/Free SSO нет в принципе — тогда это не SSO-003, а понижение тарифа).
2. Настройки → Домены: статус `pending` / `failed` вместо `verified`.
3. DNS: `dig +short TXT example.ru` — есть ли запись `yuko-verify=<токен>`.
4. Типовые ошибки в DNS: запись повешена на `www.` вместо корня; токен от другой организации/старой попытки; кавычки съедены панелью регистратора; большой TTL (запись есть, но резолвер отдаёт старый ответ).
5. Metadata IdP загружены, ACS URL и Entity ID совпадают с указанными в кабинете.

**Чинить (порядок для клиента):**
1. Настройки → Домены → скопировать TXT-значение `yuko-verify=<токен>`.
2. Добавить TXT на корень домена (host `@`), TTL поменьше (300).
3. Ждать распространение: 15–60 мин, до 24 ч при большом TTL.
4. «Проверить сейчас» в кабинете → статус `verified` → SSO поднимается в течение минуты.
5. **Всё это время вход по логину/паролю доступен** — сообщить клиенту сразу, это снимает срочность.
6. Пользователям, заведённым через SCIM без пароля, — «Забыли пароль?» для установки первого пароля.

**Эскалация:** запись видна в `dig` больше 2 часов, а кабинет пишет `failed` — тикет на платформу, приложить вывод `dig` и ID организации.

## API-401 — ключ отозван

**Симптом:** интеграция, работавшая месяцами, начала отдавать 401.

**Проверить:**
1. Интеграции → API-ключи: статус ключа. `revoked` — кто и когда (audit log).
2. **Ключ отзывается автоматически при деактивации выпустившего его пользователя** — классика после увольнения сотрудника.
3. Ключ с TTL истёк.
4. Заголовок именно `X-API-Key`, не `Authorization: Bearer`.
5. Среда: `sk_test_…` только на sandbox.
6. Обрезка/пробелы при копировании — сравнить длину и префикс.

**Чинить:** выпустить новый ключ от «технического» пользователя (не от живого сотрудника — чтобы не отозвался при увольнении), подставить в интеграцию, старый отозвать. Ключ показывается один раз при создании.

## API-403 — прав недостаточно

**Проверить:** тариф (Free → API закрыт целиком), scope ключа (`read` vs `write`), роль владельца ключа, IP-allowlist на ключе.

**Чинить:** перевыпустить ключ с нужным scope / от пользователя с нужной ролью; добавить IP в allowlist; поднять тариф до Pro.

---

## API-429 — rate limit

**Механика:** лимит на организацию: **Pro — 60 req/min, Enterprise — 600 req/min**. Ответ содержит `Retry-After` и `X-RateLimit-Remaining`.

**Проверить:** сколько ключей и клиентов бьют одновременно (лимит общий); нет ли поллинга статусов в цикле; нет ли ретрая без бэкоффа (усиливает шторм).

**Чинить:** экспоненциальный бэкофф + jitter; уважать `Retry-After`; батч-эндпоинты; заменить поллинг статусов на вебхуки `application.status_changed`. Enterprise — временное повышение лимита через менеджера.

## WH-005 — вебхук в статусе failing

**Механика:** 5 неуспешных попыток (1, 2, 4, 8, 16 мин) → эндпоинт `failing`, доставка остановлена.

**Проверить:** журнал доставки — код ответа и тело; отвечает ли приёмник 2xx за <10 сек; валидный ли TLS; правильно ли считается `X-Signature` (HMAC-SHA256 по **сырому** телу).

**Чинить:** починить приёмник (принимать в очередь, отвечать 200 сразу) → «Возобновить» → «Переотправить» пропущенные события (доступно 7 дней).

## BILL-002 — платёж отклонён

**Проверить:** срок действия карты, лимит на интернет-операции, 3-D Secure, средства.

**Чинить:** другая карта или оплата по счёту (Биллинг → «Оплатить по счёту»). Доступ включается по факту поступления, обычно 1 рабочий день; платёжка в поддержку ускоряет. Grace-период после неуспешного списания — 5 дней, дальше режим только чтение.

---

## LIM-001 — исчерпан лимит заявок

**Проверить:** тариф и счётчик на дашборде. Free — **3 заявки/мес**, обнуление 1-го числа 00:00 МСК; списание при переводе `draft` → `submitted`; удаление заявки лимит не возвращает.

**Чинить:** ждать 1-го числа или перейти на Pro (50/мес) / Enterprise (безлимит). Повышение тарифа действует немедленно, лимит доступен сразу.

## Общий алгоритм для обращения «не работает авторизация»

1. Взять из CRM: тариф, `sso_enabled`, `domain_verified`, `twofa_enabled`, роль.
2. Если `sso_enabled: true` и `domain_verified: false` → это **SSO-003**. Ответ: SSO не поднимется без DNS TXT, временно — логин/пароль, дать шаги верификации.
3. Если тариф понижен с Enterprise → SSO отключён по тарифу, вход только по паролю.
4. Если 2FA включена и жалоба «код не подходит» → AUTH-002, время на телефоне.
5. Если «слишком много попыток» → AUTH-005, ждать 15 минут, искать скрипт со старым паролем.
6. Иначе → AUTH-001, сброс пароля.
