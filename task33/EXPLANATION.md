# Объяснение task33 — разбор решения

## Что мы сделали (одним абзацем)

Представь опытного инженера поддержки. Ему приходит вопрос «не работает
авторизация». Плохой сотрудник зачитает клиенту статью из FAQ: «проверьте
пароль, проверьте раскладку». Хороший — сначала откроет карточку клиента,
увидит «SSO включён, а домен не подтверждён», и сразу скажет: вот твоя причина,
вот что делать. Мы собрали второго. Ассистент берёт вопрос, поднимает CRM
(тикет + клиент), ищет статьи в базе знаний и пишет ответ: диагноз → шаги →
что уточнить. И делает это без интернета.

## Разбор терминов (своими словами + пример из жизни)

| Термин         | Простыми словами                               | Пример из жизни                            |
| -------------- | ---------------------------------------------- | ------------------------------------------ |
| **RAG**        | Сначала загляни в справочник, потом отвечай    | Врач читает карту пациента перед диагнозом |
| **чанк**       | Кусочек документа размером с одну мысль        | Один параграф инструкции, а не вся книга   |
| **эмбеддинг**  | Текст, превращённый в числа, чтобы сравнивать  | Штрихкод: похожие товары — похожие коды    |
| **TF-IDF**     | Ищем по редким, а значит важным словам         | «SSO» важнее, чем «и», «в», «как»          |
| **косинус**    | Насколько два текста «смотрят в одну сторону»  | Два человека тычут пальцем в одну книгу    |
| **CRM**        | Картотека: кто клиент и что у него сломалось   | Карта пациента в поликлинике               |
| **MCP**        | Розетка, куда Claude втыкает инструменты       | USB: воткнул мышку — она работает          |
| **LLM-каскад** | Не вышло у первого — пробуем второго, третьего | Фонарик, свечка, спички — свет будет       |

## Почему TF-IDF, а не «настоящие» эмбеддинги

- **Ноль зависимостей.** TF-IDF — это `math` + `collections`, то есть stdlib.
  Проект запускается на голом `python3` где угодно: ни `pip install`, ни
  скачивания модели на сотни мегабайт, ни GPU.
- **Работает офлайн и мгновенно.** База знаний — 3 документа и ~50 чанков.
  Векторизация занимает миллисекунды. Тащить сюда `sentence-transformers` +
  `torch` — это 2 ГБ зависимостей ради задачи, которую решают 40 строк.
- **Терминология поддержки — редкие слова.** Клиенты пишут «SSO», «429»,
  «TXT-запись», «AUTH-005». Это ровно тот случай, где точное совпадение слова
  работает отлично: коды ошибок не нуждаются в понимании синонимов.
- **Путь наверх оставлен.** В `search.py` есть `DenseBackend`: `EMB_BACKEND=dense`
  переключает на sentence-transformers, если база знаний вырастет и понадобится
  понимать «не пускает в кабинет» ≈ «ошибка авторизации». Интерфейс `search()`
  тот же — остальной код менять не надо.

**Где TF-IDF ломается:** синонимы и перефразировки. Вопрос «не могу зайти» и
статья «ошибка аутентификации» не имеют общих слов — TF-IDF их не свяжет.
Мы это частично лечим тем, что подмешиваем в запрос тему тикета, код ошибки и
область продукта: даже если клиент написал «всё сломалось», в запрос попадёт
`SSO-003` и `auth`, и нужная статья найдётся.

## Две неочевидные вещи, которые пришлось починить

1. **Заголовки секций были невидимы для поиска.** Заголовок хранился в
   метаданных чанка, а искали только по тексту. В итоге вопрос «как выгрузить
   документы» не находил статью, буквально озаглавленную «Как выгрузить
   документы?». Лечение: заголовок секции кладём в текст чанка.
2. **Чанки-«обложки» выигрывали поиск.** Кусок из одного заголовка `# FAQ`
   (30 символов) обгонял по косинусу содержательную статью: косинус нормирует
   на длину, и короткому чанку хватало одного совпавшего слова. Такие чанки
   без тела теперь отбрасываются при индексации.

Обе — типичные грабли RAG: retrieval молча возвращает мусор, никто не падает.

## Почему каскад LLM, а не просто LLM

Каскад: **DeepSeek → Ollama → офлайн-ветка**.

- **LLM не должна быть единственной опорой.** Ключа нет, сеть отвалилась, API
  вернул 401, кончились деньги на балансе — ассистент поддержки обязан
  ответить всё равно. Демо на защите не должно зависеть от Wi-Fi в аудитории.
- **Офлайн-ветка — не заглушка, а полноценный ответ.** Это правила по данным
  CRM: если `sso_enabled=true` и `domain_verified=false` и тема auth → диагноз
  «домен не верифицирован» + 6 шагов. Плюс правила для Free+API, AUTH-005,
  API-429. Не сработало ни одно правило — берётся лучший чанк базы знаний.
- **Правила бьют точнее LLM.** Правило видит конкретные флаги конкретного
  клиента и не может нафантазировать. LLM даёт живой язык и связность —
  это усиление, а не фундамент.
- **Честность вывода.** В подвале ответа всегда написано, кто отвечал:
  `Генератор: deepseek / ollama / offline`. Видно, что именно ты читаешь.

Проверено: с битым ключом `DEEPSEEK_API_KEY=sk-bogus` каскад ловит 401,
пробует Ollama, не находит и отдаёт офлайн-ответ. Код выхода 0.

## Как контекст тикета меняет ответ

Один и тот же вопрос — **«Почему не работает авторизация?»**

**Без тикета** (`python3 support.py -q "Почему не работает авторизация?"`):

```
**Диагноз.** Похоже, ваш вопрос закрывается статьёй базы знаний
«1. Почему не работает авторизация / не могу войти?» (faq.md).

**Шаги решения:**
1. Шесть типовых причин, проверять в этом порядке:
```

Это уровень «почитайте FAQ». Ассистент не знает, кто спрашивает, и честно
предлагает общий список причин.

**С тикетом** (`python3 support.py --ticket T-1042 -q "Почему не работает авторизация?"`):

```
**Диагноз.** Игорь, SSO не активен: домен компании не верифицирован. SAML-вход
остаётся выключенным, пока в DNS домена example-import.ru не подтверждена
TXT-запись. Именно поэтому вход через SSO не проходит (код ошибки SSO-003).

**Шаги решения:**
1. Временно входите по логину и паролю — этот способ работает, SSO его не блокирует.
2. Откройте «Настройки» → «Домены компании» и скопируйте выданное значение TXT-записи.
...
```

Разница — не в формулировках, а в **знании**. Из тикета T-1042 пришли
`error_code=SSO-003` и `product_area=auth`, из карточки клиента U-100 —
`plan=Enterprise`, `sso_enabled=true`, `domain_verified=false` и почта на домене
`example-import.ru`. Шесть гипотез схлопнулись в одну причину, названную по
имени. Плюс первым шагом идёт то, что нужно клиенту прямо сейчас: как людям
работать сегодня, пока DNS обновляется.

Это и есть смысл упражнения: **RAG отвечает на вопрос, CRM отвечает на вопрос
этого клиента.**

## Что ещё могло быть (альтернативы)

| Вариант                             | Плюс                           | Минус                                        |
| ----------------------------------- | ------------------------------ | -------------------------------------------- |
| Только LLM с промптом               | Живой язык                     | Падает без сети, выдумывает флаги клиента    |
| Только FAQ-поиск                    | Просто, честно                 | «Почитайте статью» — клиент это и сам умеет  |
| Только правила (if/else)            | Точно, быстро                  | Не масштабируется, знает лишь зашитые случаи |
| **Наш гибрид (CRM + RAG + каскад)** | Работает всегда, знает клиента | Больше кода, правила надо поддерживать       |
| Векторная БД (Qdrant)               | Масштаб на миллионы статей     | Поднимать сервер ради 50 чанков — избыточно  |
| Дообученная модель                  | Знает продукт «изнутри»        | Дорого, устаревает с каждым релизом FAQ      |

## Границы решения (где честно не работает)

- **Правил всего четыре.** SSO/домен, Free+API, AUTH-005, API-429. Пятый
  сценарий уйдёт в общий RAG-ответ. Правила — это ручной труд, они не растут сами.
- **Тематика определяется по ключевым словам.** `_topic()` ищет подстроки
  («sso», «api», «вход»). Клиент, написавший «не пускает в личный кабинет»,
  в тему `auth` не попадёт.
- **RAG не понимает синонимов** (см. выше про TF-IDF).
- **База знаний — только `docs/`.** `chunk_dir` индексирует `docs/*.md`
  (product / faq / troubleshooting) и не трогает `README.md`. В task31.1
  README **был** базой знаний — ассистент отвечал на вопросы о самом проекте;
  здесь он описывает пайплайн для разработчика, и в индексе ему не место:
  иначе ассистент цитирует в «Источниках» секцию `README.md :: Пример вывода`,
  то есть собственный пример ответа. Обратная сторона: на вопросы про сам
  task33 («как устроен каскад?») RAG не ответит — это не база знаний продукта.
- **CRM — статический JSON.** Реальная поддержка ходила бы в живую базу с
  правами доступа и PII; здесь это учебный файл, который читается целиком.
- **Ассистент не пишет в CRM.** Только читает: не меняет статус тикета,
  не отвечает клиенту сам. Это осознанно — первая линия предлагает, человек решает.
- **Офлайн-ветка не умеет в диалог.** Она отвечает на один вопрос, не помнит
  предыдущий ход переписки (LLM-ветка контекст переписки видит).

## Итог одной фразой

Ассистент поддержки, который перед ответом смотрит в карточку клиента: тот же
вопрос про авторизацию превращается из «вот вам список из шести причин» в
«Игорь, у вас не подтверждён домен, вот пять шагов» — и работает даже без
нейросети и без интернета.
