# Объяснение task34 — разбор решения

## Что мы сделали (одним абзацем)

Представь нового сотрудника в первый рабочий день. Плохому надо диктовать:
«открой вторую папку слева, найди страницу двенадцать, вычеркни третий абзац».
Хорошему говоришь: «инструкция устарела, приведи её в порядок» — и он сам идёт
смотреть, как всё устроено на самом деле, сам решает, что переписать, а перед
тем как порвать старый лист, показывает: вот что уберу, вот что напишу, добро?
Мы собрали второго. Ассистент получает **цель** («приведи доку в соответствие с
кодом»), сам выбирает какие файлы открыть, что в них искать и когда
остановиться. И делает это без интернета, без API-ключей и без единой внешней
библиотеки.

Рабочий стол ассистента — папка `project/`: учебная библиотека «yuko-sdk»
(клиент к API, обработчики, отчёты, утилиты, документация, файл правил).
За пределы этой папки он выйти не может физически.

## Разбор терминов (своими словами + пример из жизни)

| Термин                | Простыми словами                                                     | Пример из жизни                                              |
| --------------------- | -------------------------------------------------------------------- | ------------------------------------------------------------ |
| **MCP**               | Общий стандарт розетки, куда втыкаются инструменты для ИИ            | Евророзетка: любой прибор любого завода — воткнул и работает |
| **JSON-RPC**          | Бланк заявки строгой формы: что сделать, с чем, и куда ответ         | Талон в МФЦ: графа «услуга», графа «данные», графа «номер»   |
| **stdio**             | Разговор через окошко: сунул бумажку — получил бумажку               | Окошко в кассе: подал листок, забрал листок. Без интернета   |
| **ReAct**             | Подумал → сделал одно действие → посмотрел что вышло → снова подумал | Ищешь ключи: заглянул в карман, не там → теперь в куртку     |
| **tool (инструмент)** | Одно конкретное умение, которое можно попросить выполнить            | Ящик с инструментами: молоток, отвёртка. Берёшь нужное       |
| **observation**       | Что получилось после действия. Факт, а не догадка                    | Открыл холодильник — увидел, что молока нет. Это observation |
| **песочница**         | Огороженная площадка: играй сколько хочешь, за забор — нельзя        | Ребёнок во дворе: калитка заперта, гулять только внутри      |
| **unified diff**      | Список правок: минус — что убираю, плюс — что добавляю               | Правки редактора красной ручкой на полях рукописи            |
| **dry-run**           | Репетиция: всё показываю, но по-настоящему не делаю                  | Примерка в магазине: надел, посмотрел, ничего не купил       |
| **ast**               | Разбор текста программы так, как его читает сам Python               | Разбор предложения по членам: где подлежащее, где сказуемое  |
| **идемпотентность**   | Сделал второй раз — ничего не изменилось                             | Нажал кнопку лифта дважды — лифт не приедет дважды           |
| **LLM-каскад**        | Не вышло у первого — пробуем второго, потом третьего                 | Фонарик, свечка, спички — свет будет в любом случае          |

## Из чего собран ассистент

Пять файлов, у каждого одна ответственность.

| Файл | Что делает |
| --- | --- |
| `fs_tools.py` | Руки: `fs_list` / `fs_read` / `fs_grep` / `fs_write` / `fs_diff` + песочница `_safe()` |
| `mcp_fs_server.py` | Те же пять инструментов, но по MCP — чтобы ими пользовался Claude Desktop |
| `agent.py` | Мозг: ReAct-цикл, `ast`-разбор, генераторы markdown, офлайн-планы |
| `llm.py` | Каскад DeepSeek → Ollama → офлайн |
| `main.py` | CLI: `--goal`, `--dry-run`, `--max-steps`, `--out` |

Инструменты написаны **один раз** в `fs_tools.py` и обёрнуты дважды: для агента
(`agent.TOOLS`) и для MCP (`mcp_fs_server.TOOLS`). Одна кухня — два окна выдачи.

## Что получилось на живых прогонах

| Сценарий                         | Что нашёл                                                                                                                                            |
| -------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1. Поиск использования           | 8 упоминаний в 3 файлах. Определён `api_client.py:11`; импортируют `handlers.py` и `reports.py`; создают экземпляр `handlers.py:36`, `reports.py:17` |
| 2. Обновление доки (`--dry-run`) | 4 класса расхождений (см. ниже). Diff показан, файл не тронут                                                                                        |
| 3. Генерация README              | Разобрано 4 модуля из 8 файлов → `project/README.md`, 75 строк. Повтор → «изменений нет»                                                             |
| 4. Проверка инвариантов          | 3 нарушения: `ApiClient.post`, `ApiClient.close`, `reports.export_csv` — без docstring                                                               |

Четыре расхождения из сценария 2 — это, собственно, ради чего всё:

1. Дока описывает метод `fetch()`, **которого в коде нет**. Рассказ про то, чего не существует.
2. Реальные `close()` и `retry_count` в доке **не описаны вообще**.
3. Дока обещает `ApiClient(base_url, token, retries=3)`. В коде — `__init__(self, base_url, token, timeout=30)`. **Дока врёт про параметр**: напишешь `retries=3` — получишь `TypeError`. Повторы живут в свойстве `retry_count`.
4. `post` и `close` — без docstring. Ассистент не выдумал за автора описание, а честно написал в новой доке: «Docstring отсутствует в коде — нарушение CONVENTIONS.md». Чинить надо в коде.

## Почему именно так

### Почему MCP, а не свой протокол

Свой протокол — это розетка собственной конструкции. Работает, но вилку к ней
никто не подберёт. MCP — стандарт: пропиши сервер в `claude_desktop_config.json`
одним блоком, перезапусти Claude Desktop — и он сам ходит в папку `project/`
этими пятью инструментами. Ноль кода на стыковку.

При этом **MCP-сервер написан на чистом stdlib**, без SDK: реализованы
`initialize`, `tools/list`, `tools/call`, `notifications/initialized`, ошибки
отдаются кодом `-32602`, как положено. Причина простая: сервер должен
запускаться на голом `python3`, без `pip install`. Требование «ноль
зависимостей» — не поза, а условие воспроизводимости.

### Почему `ast`, а не регулярки

Регулярка `def (\w+)\(` — это чтение по верхам. Она сломается на многострочной
сигнатуре, на слове `def` внутри строки, на декораторах, на вложенных функциях.
`ast` — тот самый парсер, которым Python читает файл перед запуском: если код
валиден, разбор точен по определению.

Практический эффект виден в результате: свойство `retry_count` попало в новую
доку **не потому, что кто-то его туда вписал**, а потому что оно лежит в дереве
разбора с декоратором `@property`. Дока физически не может разъехаться с кодом —
она из кода и растёт. Один разборщик `analyze_source()` обслуживает все три
сценария: генерацию доки, генерацию README и проверку правил.

### Почему офлайн-планы

Домашку должен воспроизвести проверяющий — не ища API-ключ и не поднимая
Ollama. Живая LLM каждый прогон ходит немного разной дорогой: другое число
шагов, другие формулировки. Проверять такое нечем.

Важно: офлайн — это **не «вместо агента»**. `FileAgent` один и тот же ReAct-цикл
в обоих режимах.

| | LLM-режим | Офлайн-режим |
| --- | --- | --- |
| План шагов | строит модель | зашит в `offline_plan()` |
| Инструменты | `TOOLS` | те же `TOOLS` |
| Протокол шагов, лог, отчёт | одинаковые | одинаковые |
| Итог | пишет модель | собирает код |
| Воспроизводимость | нет | побайтная |

Меняется **только источник плана**. `export DEEPSEEK_API_KEY=…` — и та же
команда уйдёт в модель. Плюс каскад страхует: если сеть отвалилась посреди
работы, агент допишет протокол и перейдёт на офлайн-план, а не упадёт.

### Почему песочница

Модель может попросить что угодно — в том числе `../../etc/passwd`. `_safe()` —
единственная дверь к файловой системе: любой путь резолвится относительно
`ROOT = project/`, и если результат оказался снаружи — `ValueError`.

Ключевая деталь: проверка идёт **по уже резолвленному пути**. Поэтому ловятся
не только `../`, но и абсолютные пути, и симлинки наружу. Проверять исходную
строку («а нет ли в ней двух точек?») — классическая дыра: обойти такую защиту
можно десятком способов.

### Почему diff, а не сразу запись

Потому что ассистент, который молча переписал файл, — это не помощник, а
рулетка. `fs_write` **всегда сначала считает diff**, и только потом решает
писать. Три уровня защиты:

1. **Песочница** — за `project/` не выйдет никогда.
2. **`--dry-run` — защита агента, а не вежливая просьба к модели.** В `FileAgent._call_tool()` флаг `dry_run=True` проставляется принудительно любому `fs_write`. Что бы модель ни попросила, записи не будет. Защита на уровне кода, а не на уровне промпта — промпт модель может проигнорировать, а `if` не может.
3. **Diff перед записью** — видно, что изменится, до того как оно изменится.

Побочный, но приятный эффект: если diff пустой — на диск не идём **вообще**. Не
«перезаписали тем же самым», а не тронули. Отсюда идемпотентность: второй прогон
генерации README говорит «изменений нет» и не создаёт шумного коммита.

### Почему цель, а не прямая команда

В этом весь смысл упражнения. «Открой файл X, замени строку 12» — тут думает
человек, а ассистент работает руками. Ценность нулевая: если ты уже знаешь, что
на строке 12, ты и сам её поправишь.

«Приведи доку в соответствие с кодом» — тут человек знает **цель**, а решение
ищет ассистент. И это видно в логе сценария 4: сначала прочитал `CONVENTIONS.md`
(узнать, какие вообще правила проверять), потом `fs_list` (какие есть модули),
потом `fs_grep` по `print()`, потом чтение всех четырёх модулей, потом отчёт.
Порядок осмысленный: правила → файлы → проверка. Никто не прописывал «читай
CONVENTIONS первым».

Там же самый показательный момент: ассистент отдельно отмечает, что приватный
`_request` правило не затрагивает — потому что **в `CONVENTIONS.md` так
написано**. Он прочитал правило и применил его, а не отработал зашитый список.

## Что ещё могло быть (альтернативы)

| Вариант | Плюс | Минус |
| --- | --- | --- |
| Прямые команды («правь строку 12») | Предсказуемо | Думает человек, ассистент не нужен |
| Регулярки вместо `ast` | Пишется за 5 минут | Ломается на декораторах, многострочных сигнатурах, `def` в строке |
| Отдать файлы модели целиком, без инструментов | Просто | Контекст переполнится, модель выдумает содержимое |
| Патч по строкам вместо записи целиком | Экономно | Самое хрупкое место: модель промахнётся на строку и молча испортит файл |
| **Наш вариант (цель + tools + `ast` + diff)** | Работает всегда, ничего не портит | Больше кода, офлайн-планы надо поддерживать |
| Свой протокол вместо MCP | Полная свобода | Никто не подключится, стыковку писать руками |
| MCP через официальный SDK | Меньше кода | `pip install`, а домашка должна идти на голом `python3` |
| Только LLM, без офлайн-ветки | Гибко, любые цели | Нет ключа → демо мертво. Каждый прогон разный, проверять нечем |
| Только офлайн-планы, без LLM | Побайтно воспроизводимо | Знает лишь 4 сценария, ничего нового не придумает |

Отдельно про **запись файла целиком**. Кажется расточительным — но патч по
строкам в таких инструментах ломается чаще всего: модель ошиблась на одну
строку, и файл испорчен без единого сообщения об ошибке. Запись целиком +
обязательный diff даёт полный контроль: либо новый файл целиком правильный,
либо в diff сразу видно, что он кривой. Для файлов такого размера — разумный
размен.

## Границы решения (где честно не работает)

- **Офлайн-сценариев ровно четыре.** `detect_scenario()` считает взвешенные ключевые слова. Точной фразы не требует — «обнови документацию» и «актуализируй api.md» ведут в один план. Но цель вне четвёрки уйдёт в `_plan_fallback()`: ассистент честно скажет, что готового плана нет, и покажет, что умеет. Не притворяется.
- **Проверяются три правила из `CONVENTIONS.md`**, а не все: docstring у публичных членов, `print()` в библиотечном коде, `snake_case` в именах модулей. «Длина строки 88» и «порядок импортов» не проверяются.
- **`ast` понимает только Python.** Markdown, YAML, конфиги разбираются как текст.
- **`fs_grep` ограничен 200 хитами**, `observation` обрезается до 4000 символов — иначе большой файл съест весь контекст модели. На больших проектах это заметно.
- **Ассистент не запускает код**, только читает и пишет. Проверить, что после его правки всё ещё работает, он не может — это делает человек.
- **`docs/api.md` устарела намеренно** — ровно та ситуация, ради которой всё делалось. Код в `project/` не сломан: он рабочий, просто у двух методов забыли docstring. Как в жизни.

## Итог одной фразой

Ассистент, которому ставят цель, а не диктуют шаги: он сам решил, что `fs_grep`
быстрее чтения всех файлов, что `CONVENTIONS.md` надо прочитать раньше кода и
что код — источник правды для доки, а не наоборот; при этом за пределы своей
папки не выйдет, ничего не перепишет без показанного diff, а разбор через `ast`
не даёт документации разъехаться с кодом — и всё это работает без нейросети и
без интернета.
