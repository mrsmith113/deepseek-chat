# task34 — Мозг ассистента: ReAct-агент над файлами проекта

Ассистент, который получает **цель**, а не команду. Куда смотреть, что искать,
что править и когда остановиться — решает сам, вызывая файловые инструменты.

Разница принципиальная:

| Обычный скрипт | Этот ассистент |
|---|---|
| «открой `docs/api.md`, замени строку 12» | «приведи `docs/api.md` в соответствие с кодом» |
| человек знает решение и диктует шаги | человек знает **цель**, шаги ищет ассистент |

Работает **полностью офлайн**: без API-ключей, без интернета, без внешних
зависимостей. Только стандартная библиотека Python 3.10+.

## Архитектура

```
                       ┌──────────────────────────────┐
   ЦЕЛЬ (--goal)  ───► │        FileAgent.run()       │
                       └───────────────┬──────────────┘
                                       │
                       ┌───────────────▼──────────────┐
                       │  llm.chat()  — LLM-каскад    │
                       │  DeepSeek → Ollama → None    │
                       └───────┬──────────────┬───────┘
                     есть LLM  │              │  None
                               ▼              ▼
                    ┌──────────────────┐  ┌──────────────────────┐
                    │  ReAct-цикл      │  │  offline_plan(goal)  │
                    │  модель решает   │  │  план зашит в код    │
                    └────────┬─────────┘  └──────────┬───────────┘
                             │                       │
                             └───────────┬───────────┘
                                         ▼
                        ┌────────────────────────────────┐
                        │  TOOLS: fs_list · fs_read ·    │
                        │  fs_grep · fs_diff · fs_write  │
                        └────────────────┬───────────────┘
                                         ▼
                        ┌────────────────────────────────┐
                        │  fs_tools._safe() — песочница  │
                        │       только project/          │
                        └────────────────┬───────────────┘
                                         ▼
                              observation (≤4000 симв)
                                         │
                        ┌────────────────▼───────────────┐
                        │  назад в историю → новый шаг   │
                        │  пока не final_answer / limit  │
                        └────────────────────────────────┘
```

### Шаг ReAct-цикла

Модель на каждом шаге отвечает **строго одним JSON-объектом**:

```json
{"thought": "нужно понять, кто зовёт клиент", "action": "fs_grep",
 "action_input": {"pattern": "ApiClient", "glob": "*.py"}}
```

либо завершает работу:

```json
{"thought": "фактов достаточно", "final_answer": "ApiClient определён в api_client.py…"}
```

Результат инструмента возвращается как `observation`, обрезается до 4000
символов и уходит в историю. Дальше — следующий шаг, и так до `final_answer`
или до `--max-steps` (по умолчанию 12).

Парсер ответа устойчив к типичному поведению моделей: вырезает ```json-заборы,
находит первый сбалансированный `{…}` с учётом строк и экранирования, а на
неразобранный ответ просит модель повторить в правильном формате.

### Три уровня защиты

1. **Песочница.** `fs_tools._safe()` резолвит любой путь относительно
   `project/` и бросает `ValueError` при попытке выйти наружу — `../../etc/passwd`,
   абсолютный путь и симлинк наружу не пройдут.
2. **`--dry-run` — защита агента, а не опция модели.** В этом режиме
   `FileAgent._call_tool()` принудительно проставляет `dry_run=True` любому
   `fs_write`. Что бы модель ни попросила, записи на диск не будет — вернётся
   только diff.
3. **Diff перед записью.** `fs_write` всегда считает unified diff: видно, что
   именно изменится, ещё до изменения.

## Четыре сценария

Все четыре работают без LLM: `offline_plan(goal)` подбирает по ключевым словам
цели детерминированную цепочку вызовов, а итог собирается кодом.

### 1. Поиск использования API

```bash
python3 main.py --goal "найди все места где используется ApiClient"
```

`fs_grep` по `ApiClient|api_client` → группировка по файлам → `file:line` с
ролью каждой строки (определение / импорт / создание экземпляра) и сводка:
где определён, кто импортирует, кто вызывает.

### 2. Обновление доки по коду

```bash
python3 main.py --goal "приведи docs/api.md в соответствие с кодом api_client.py" --dry-run
```

`fs_read api_client.py` → разбор `ast` (публичные методы, сигнатуры, docstring,
property, константы) → `fs_read docs/api.md` → генерация нового markdown →
`fs_write` с diff. Ассистент явно называет расхождения: несуществующий `fetch()`,
неописанные `close` и `retry_count`, выдуманный параметр `retries=3`.

Без `--dry-run` файл будет перезаписан.

### 3. Генерация нового файла

```bash
python3 main.py --goal "сгенерируй README.md для проекта"
```

`fs_list` → `fs_read` каждого `.py` → `ast`: модули, классы, функции,
docstring → сборка `project/README.md` (описание, структура, публичный API,
установка, использование) → `fs_write`.

### 4. Проверка инвариантов

```bash
python3 main.py --goal "проверь соответствие кода правилам из CONVENTIONS.md"
```

`fs_read CONVENTIONS.md` → `fs_list` → обход `ast` + `fs_grep`. Три правила:
публичные функции и методы без docstring, `print()` в библиотечном коде,
имена модулей не в `snake_case`.

Находит **3 нарушения**: `ApiClient.post`, `ApiClient.close`,
`reports.export_csv` — все без docstring.

### Флаги

```
--goal, -g     цель (без неё — список демо-целей)
--dry-run      не писать на диск, показать только diff
--max-steps N  лимит шагов агента (по умолчанию 12)
--out FILE     markdown-отчёт: цель, шаги, diff'ы, итог
--quiet, -q    только итог, без протокола шагов
```

Всё сразу:

```bash
bash run.sh
```

## Почему офлайн работает

Ключевой момент домашки — **воспроизводимость**. Проверяющий не должен искать
API-ключ или поднимать Ollama, чтобы увидеть результат.

`llm.chat()` возвращает `None`, если нет ни DeepSeek, ни Ollama. Это не ошибка,
а штатная ветка: агент переходит на `offline_plan(goal)` — заранее прописанную
цепочку тех же самых вызовов инструментов. Меняется только источник плана:

| | LLM-режим | Офлайн-режим |
|---|---|---|
| План шагов | строит модель | зашит в `offline_plan()` |
| Инструменты | `TOOLS` | те же `TOOLS` |
| Протокол шагов | одинаковый | одинаковый |
| Итог | пишет модель | собирает код |
| Воспроизводимость | нет | побайтная |

Сценарий выбирается взвешенным подсчётом ключевых слов (`detect_scenario`),
поэтому цель можно формулировать свободно — «обнови документацию» и
«актуализируй api.md» ведут в один план. Цель не распознана — ассистент честно
скажет, что готового плана нет, и покажет доступные.

Разбор Python везде через `ast`, не регулярками: сигнатуры, docstring,
декораторы и property берутся из дерева разбора, поэтому дока не может
разъехаться с кодом.

`run.sh` **идемпотентен**: повторный прогон даёт то же состояние файлов.
Сценарий 2 идёт с `--dry-run`, сценарий 3 генерирует README без самого себя
в списке файлов, поэтому второй запуск даёт «изменений нет» вместо нового diff.

## Подключение MCP-сервера к Claude Desktop

`mcp_fs_server.py` — те же инструменты по Model Context Protocol (JSON-RPC over
stdio), на чистом stdlib и без установки SDK.

`claude_desktop_config.json`:

* macOS: `~/Library/Application Support/Claude/claude_desktop_config.json`
* Windows: `%APPDATA%\Claude\claude_desktop_config.json`

```json
{
  "mcpServers": {
    "project-files": {
      "command": "python3",
      "args": ["/абсолютный/путь/к/task34/mcp_fs_server.py"]
    }
  }
}
```

Перезапустить Claude Desktop — в списке инструментов появятся `fs_list`,
`fs_read`, `fs_grep`, `fs_write`, `fs_diff`, работающие в песочнице `project/`.

Проверка вручную, без Claude:

```bash
echo '{"jsonrpc":"2.0","id":1,"method":"tools/list"}' | python3 mcp_fs_server.py

echo '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"fs_grep","arguments":{"pattern":"class ","glob":"*.py"}}}' | python3 mcp_fs_server.py
```

## Опциональная LLM

Офлайн — режим по умолчанию. Чтобы план строила модель:

```bash
export DEEPSEEK_API_KEY="sk-…"        # приоритетный бэкенд
# либо локальная Ollama:
export OLLAMA_MODEL="qwen3:14b"       # по умолчанию qwen3:14b
export OLLAMA_HOST="http://localhost:11434"

python3 main.py --goal "разберись, почему reports.py не закрывает клиент"
```

Активный бэкенд печатается в начале каждого прогона (`llm.llm_available()`).
Если сеть отвалится посреди работы, агент допишет протокол и не упадёт.
