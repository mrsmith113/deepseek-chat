# Task 14 — Инварианты и ограничения состояния

## Что сделано

Добавлена система инвариантов — жёстких ограничений, которые агент не имеет права нарушать.

---

## Архитектура

```
invariants.py          — новый модуль управления инвариантами
hornest.py             — +импорт, +init, +роутинг /inv, +детект конфликта
memory.py              — +1 строка: build_invariant_prompt() в build_context()
hornest.db             — +таблица invariants
```

### Таблица `invariants`

```sql
CREATE TABLE invariants (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    category    TEXT NOT NULL DEFAULT 'general',
    title       TEXT NOT NULL,
    description TEXT NOT NULL,
    created_at  TEXT
)
```

Категории: `architecture` / `stack` / `business` / `security` / `general`

---

## Как работает

### 1. Хранение
Инварианты лежат в SQLite отдельно от диалога (`messages`, `long_term_memory`).  
Это не "воспоминания" — это жёсткие правила.

### 2. Инжекция в промпт
`build_invariant_prompt()` вызывается внутри `build_context()` **перед каждым запросом**.  
Блок инвариантов вставляется как системное сообщение сразу после профиля пользователя.

Формат блока:
```
══════════════════════════════════════════════
⛔ ИНВАРИАНТЫ — ЖЁСТКИЕ ОГРАНИЧЕНИЯ
══════════════════════════════════════════════
Это абсолютные правила. Ты ОБЯЗАН их соблюдать.
Если пользователь просит что-то, что нарушает хотя бы один инвариант:
1. Ответь: [CONFLICT: <название инварианта>]
2. Объясни, какой инвариант нарушен и почему
3. Предложи альтернативу (если возможно)
...

🏗  Архитектура:
  [1] Только SQLite
      → Не использовать PostgreSQL, MySQL или любые внешние БД

🧱 Стек:
  [2] Python-only backend
      → Не предлагать Node.js, Go, Rust для серверной части
...
```

### 3. Детект конфликта
Если агент обнаружил нарушение, он возвращает `[CONFLICT: название]`.  
`detect_conflict(answer)` парсит это и Hornest выводит:

```
  ⛔ КОНФЛИКТ С ИНВАРИАНТОМ: Только SQLite
  ──────────────────────────────────────────────────────
  
  💻 Coder:
  [CONFLICT: Только SQLite]
  
  Я не могу порекомендовать PostgreSQL в данном контексте, 
  так как это нарушает ваш архитектурный инвариант...
```

## Команды

| Команда | Действие |
|---|---|
| `/inv` или `i` | Список всех инвариантов |
| `/inv list` | То же |
| `/inv add` | Интерактивное добавление |
| `/inv rm` | Интерактивное удаление |
| `/inv check <текст>` | Проверить текст через API |

## Примеры инвариантов для проекта Hornest

### Архитектура
- `Только SQLite` — не использовать внешние БД
- `Монолит` — не разбивать на микросервисы, пока не будет >1000 DAU
- `CLI-first` — интерфейс только терминальный, без web/GUI

### Стек
- `Python-only` — не предлагать другие языки для бэкенда
- `Без фреймворков` — requests и stdlib, не FastAPI/Flask
- `DeepSeek API` — не переключать на OpenAI/Anthropic API

### Бизнес-правила
- `Бесплатный режим` — все фичи доступны без платной подписки
- `Данные локально` — никакой отправки данных на внешние серверы кроме API

### Безопасность
- `API-ключ в .env` — не хардкодить ключи в коде
- `Нет логирования запросов` — не сохранять полные промпты в логах
