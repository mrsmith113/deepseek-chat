# Архитектура AI code review (task32)

## Назначение

Пайплайн автоматического ревью pull request. На вход — diff PR и изменённые
файлы, на выход — текст ревью с тремя разделами: потенциальные баги,
архитектурные проблемы, рекомендации.

## Поток данных

1. **Источник diff** — GitHub Action на событии `pull_request` берёт
   `git diff BASE...HEAD`. Локально можно подать готовый `.diff` файл.
2. **diff_parser** — разбирает unified diff: изменённые файлы, добавленные
   строки с номерами, язык файла.
3. **heuristics** — статический анализатор по добавленным строкам: секреты,
   широкие except, eval/exec, мутабельные дефолты, `== None`, `console.log`,
   слишком длинные строки, крупный diff. Работает офлайн.
4. **RAG** — по каждому изменённому файлу ищем в индексе (TF-IDF, косинус)
   релевантные куски документации и кода проекта. Индекс — `index.json`,
   без внешней БД.
5. **LLM-каскад** — DeepSeek API → Ollama (Qwen3) → без LLM. LLM усиливает
   ревью, но не обязательна: эвристики + RAG дают результат всегда.
6. **Рендер** — итоговый markdown, который постится комментарием в PR.

## Принципы

- **Офлайн-first**: каждый шаг работает без сети и без поднятых сервисов.
- **Одна ответственность на файл**: parser, heuristics, search, reviewer.
- **Переиспользование**: search/chunking/git_tools взяты из task31.1.
- **RAG = документация + код**: индексируем и `.md`, и `.py`.
