# Правила ревью и стандарты проекта

Эти правила использует RAG-контекст: ассистент сверяет diff с ними.

## Безопасность

- Секреты (пароли, токены, api-ключи) — только через переменные окружения,
  никогда не хардкодить в коде.
- Не использовать `eval` / `exec` на данных из вне — риск инъекции кода.
- `subprocess` — без `shell=True` на пользовательском вводе.

## Надёжность

- Ловить конкретные исключения, а не голый `except:` — иначе глотаются баги.
- Не использовать мутабельные аргументы по умолчанию (`def f(x=[])`).
- Сравнение с None — через `is None`, не `== None`.

## Стиль и архитектура

- Логирование через `logging`, а не `print` в прод-коде.
- Длина строки — до 120 символов.
- Крупные PR (150+ добавленных строк в файл) желательно дробить.
- Один модуль — одна ответственность.

## Процесс

- Каждый PR проходит автоматическое AI-ревью (GitHub Action `ai-review.yml`).
- Ревью не блокирует мерж, а подсказывает — финальное решение за человеком.
